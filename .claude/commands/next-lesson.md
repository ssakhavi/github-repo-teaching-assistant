## Step 1: Select Repository

Use the Glob tool to list all files matching `input/*.txt`.

- **No files found:** Stop and tell the user:
  > No repository files found in `input/`. Add a `.txt` file generated by git-ingest:

  > gitingest https://github.com/user/repo --output input/my-repo.txt
  > ```

- **$ARGUMENTS provided:** Use the value as the repo name (e.g., `/next-lesson fastapi` → repo = `fastapi`). Verify `input/fastapi.txt` exists.

- **Exactly one file found:** Use its stem as the repo name automatically (e.g., `input/fastapi.txt` → `fastapi`). Tell the user which repo was selected.

- **Multiple files found and no $ARGUMENTS:** List the available repos and ask the user to specify:
  > Multiple repos found in `input/`:
  > - `fastapi` → `input/fastapi.txt`
  > - `react` → `input/react.txt`
  >
  > Run `/next-lesson <repo-name>` to specify which one to use.
  >
  > Stop here.

From this point, all paths are rooted at `training/<repo_name>/`:

## Step 6: Generate Lesson Content (Three Parallel Subagents)

Launch **three subagents in parallel** using the Agent tool. Send all three Agent tool calls in a single response turn.

Substitute the actual repo name for `<repo>` and lesson number for `{N+1}` in every prompt below.

### Subagent A — Lesson Text

**Agent tool call:**
- `description`: "Generate lesson text for <repo> lesson {N+1}"
- `subagent_type`: `general-purpose`
- `prompt`:

```
You are generating the lesson text for the GitHub Training Assistant.

Repo: <repo>
Lesson number: {N+1}

Read `training/<repo>/lessons/lesson_{N+1}/blueprint.md` fully — focus on the Content Outline, Key Concepts, Code Focus, What NOT to Cover, and **User Feedback** sections. If User Feedback lists specific confusion points or topics the learner wants more of, ensure those are addressed in the lesson content where they fit within scope.

If the blueprint contains a **Learner Background** section, apply it throughout the lesson:
- Use the listed **Analogy sources** when explaining every concept — ground analogies in the user's domain, not generic examples
- Follow all **Calibration directives** to adjust depth — skip scaffolding the user doesn't need, add depth where they have relevant expertise
- Respect the **Avoid** list — do not use framing or analogies from unfamiliar domains
- Write as if speaking to someone with that professional context, not as a generic tutorial

Read `input/<repo>.txt` — specifically the files listed in the blueprint's "Code Focus" section. Be strategic; do not read the whole file.

Write a paginated lesson to `training/<repo>/lessons/lesson_{N+1}/lesson.md`:

- Pages separated by a line containing only `---`
- Minimum 4 pages, maximum 10 pages
- Page 1: Title (# heading), lesson number, repo name, what will be covered, why it matters
- Middle pages: one sub-topic per page from the Content Outline; use ## headings, real code snippets in fenced code blocks with language specifiers, explain "why" not just "what"
- Last page: ## Summary — "In this lesson we covered:" with 3–5 bullets, key takeaways, preview of next lesson
- Style: conversational, second person ("you"), ~300–500 words per page
- CRITICAL: Do NOT reference concepts listed under "What NOT to Cover" — those belong to future lessons

After writing, confirm the file path and how many pages were generated.
```

---

### Subagent B — MCQ Questions

**Agent tool call:**
- `description`: "Generate MCQ questions for <repo> lesson {N+1}"
- `subagent_type`: `general-purpose`
- `prompt`:

```
You are generating multiple choice questions for the GitHub Training Assistant.

Repo: <repo>
Lesson number: {N+1}

Read `training/<repo>/lessons/lesson_{N+1}/blueprint.md` — focus on Learning Objectives, Key Concepts, MCQ Topic Areas, Prior Knowledge Available, What NOT to Cover, and **User Feedback**. If the User Feedback section flags specific concepts as confusing, make sure those concepts appear as questions (so the learner can retest their understanding) without being the only topic covered.

If the blueprint contains a **Learner Background** section, apply it to question framing:
- Use domain-appropriate scenario framing where possible (e.g. "In a production API context..." for a backend engineer; "In a data pipeline..." for an ML practitioner)
- Use **Calibration directives** to set distractor difficulty — senior engineers get subtler wrong answers, beginners get clearer distinctions
- Do not reveal any content from the Learner Background section in question text itself

Read `input/<repo>.txt` — specifically the files in the Code Focus section. Every factual claim must be verifiable against the actual code.

Generate 5–10 questions and write them to `training/<repo>/lessons/lesson_{N+1}/mcq.json`:

{
  "lesson": {N+1},
  "title": "<title from blueprint>",
  "questions": [
    {
      "id": 1,
      "question": "Question text ending with ?",
      "options": {
        "A": "...",
        "B": "...",
        "C": "...",
        "D": "...",
        "E": "I don't know"
      },
      "correct_answer": "B",
      "explanation": "B is correct because [reason]. A is wrong because [brief]. C [brief]. D [brief]."
    }
  ]
}

Rules:
- Option E is ALWAYS "I don't know" — include it in every question verbatim
- correct_answer is one of "A", "B", "C", "D" — never "E"
- Each question tests ONE concept from lesson {N+1} or earlier lessons only — NO forward references
- Concepts listed under "What NOT to Cover" in the blueprint must not be tested
- Difficulty: ~30% easy, ~50% medium, ~20% hard
- Mix question types: factual recall, conceptual understanding, code reading, application
- Minimum 2-sentence explanations
- Valid JSON with proper string escaping

After writing, confirm the file path and number of questions.
```

### Subagent C — Coding Exercises

**Agent tool call:**
- `description`: "Generate coding exercises for <repo> lesson {N+1}"
- `subagent_type`: `general-purpose`
- `prompt`:

```
You are generating coding exercises for the GitHub Training Assistant.

Repo: <repo>
Lesson number: {N+1}

Read `training/<repo>/lessons/lesson_{N+1}/blueprint.md` — focus on Code Focus, Example Exercise Ideas, Learning Objectives, Prior Knowledge Available, What NOT to Cover, and **User Feedback**. If the User Feedback section identifies specific confusion about a concept, prioritise an exercise that directly practices that concept (within scope).

Read the specific files from `input/<repo>.txt` listed in the Example Exercise Ideas section.

Also read `training/<repo>/results/lesson_0_background.json` to check the learner's programming_experience level — calibrate difficulty accordingly (more scaffolding for beginners, denser exercises for seniors).

If the blueprint contains a **Learner Background** section, apply it to exercise design:
- Use **"Bias code examples toward"** when choosing exercise scenarios, function names, and docstring context — e.g. an ML engineer's exercises should use data-pipeline scenarios; a backend engineer's should use API/service scenarios
- Follow **Calibration directives** to skip over-explaining concepts the user already knows
- Use **"Use analogies from"** when writing the exercise description's "why this matters" framing

Generate 2–3 exercises and write them to `training/<repo>/lessons/lesson_{N+1}/examples.json`:

{
  "lesson": {N+1},
  "title": "<title from blueprint>",
  "examples": [
    {
      "id": 1,
      "type": "fill_in_blank",
      "title": "Short descriptive title",
      "description": "What this tests and why it matters",
      "template": "def my_func(x):\n    # ___BLANK___  (description of what goes here)\n    pass",
      "solution": "def my_func(x):\n    return x * 2",
      "tests": [
        {
          "code": "assert my_func(5) == 10, f'Expected 10, got {my_func(5)}'",
          "description": "Basic case returns double"
        }
      ]
    }
  ]
}

Rules:
- Exercise 1: always "fill_in_blank" (scaffolded)
- Exercise 2–3: "write_whole" or "fill_in_blank"
- CRITICAL: exercises must only use concepts from lesson {N+1} or earlier — no forward references
- Concepts under "What NOT to Cover" must not appear in exercises
- Functions must use standard library only — no imports from the actual repo
- fill_in_blank: template is valid Python with # ___BLANK___  (description) comments
- write_whole: template is function signature + detailed docstring + pass
- Tests: 2–5 per exercise; each "code" field is a complete self-contained Python snippet; tests PASS with solution, FAIL with unmodified template
- Valid JSON with proper string escaping

After writing, confirm the file path, number of exercises, and their types.
```
